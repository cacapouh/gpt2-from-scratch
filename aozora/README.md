# 明治文学で日本語 GPT-2 をファインチューニング

`rinna/japanese-gpt2-medium` (336M) を青空文庫の明治文学でファインチューンし、
現代的なプロンプトにも**明治知識人風の論説文**で答えさせる実験。

親プロジェクト（[../README.md](../README.md)）が GPT-2 をゼロから実装するのに対し、
こちらは HuggingFace Transformers を使った**実用的なファインチューン最小構成**です。

## 使い方

### 1. セットアップ

```powershell
cd aozora
# torch は親プロジェクトで入っている想定（CUDA 12.8、Python 3.12 .venv を共有）
pip install -r requirements.txt
```

> `requirements.txt` に `protobuf` を含む点に注意。`T5Tokenizer`
> （SentencePiece 変換）の読み込みに必要。

### 2. コーパス構築

`data/raw/` には青空文庫の原文 36 ファイルが同梱済み。ここから
マークアップ除去と比率連結を行い、学習用の 1 ファイルを作る。

```powershell
python main.py clean       # raw → cleaned（ルビ《》・注記［＃］・底本情報を除去）
python main.py prepare     # cleaned → combined.txt（作家別配合で連結）

# または一括実行（raw が無い / 再ダウンロードしたい場合）
python main.py data        # download → clean → prepare
```

期待値: `data/combined.txt` は約 **7,535,205 字** (seed=42, 2026-04 時点、14 作家 71 作品)。
各作品の先頭に `【作家名】` タグが挿入され、生成時の文体指定に使える。

### 3. ファインチューン

```powershell
# RTX 5070 (12GB) で 7.5M 字 × 10 epoch (stride 256) ≈ 約 24 時間、ピーク VRAM 約 10 GB
python main.py finetune --epochs 10 --batch-size 2 --grad-accum-steps 4 --stride 256 --lr 3e-5
```

実効バッチサイズ 8（= `batch-size 2 × grad-accum-steps 4`）。
コサイン LR スケジューラ (warmup 5%) と勾配クリッピング (`max_grad_norm=1.0`) がデフォルトで有効。
`--stride 256` はスライディングウィンドウを 50% オーバーラップさせ、実効データ量を 2 倍にして境界文脈の学習を強化する。

出力: `checkpoints/meiji_gpt2_v9/`（`model.safetensors` 約 1.34 GB + tokenizer）

### 4. 生成

```powershell
# ファインチューン後
python main.py generate --model checkpoints/meiji_gpt2_v9 --prompt "電子計算機とは、" `
    --temperature 0.7 --top-k 40 --top-p 0.9 `
    --repetition-penalty 1.2 --no-repeat-ngram-size 3

# 作家タグで文体を指定（先頭に `【作家名】\n\n` を付ける）
python main.py generate --model checkpoints/meiji_gpt2_v9 --prompt "【夏目漱石】`n`n吾輩は"

# 比較用（素の rinna）
python main.py generate --model rinna/japanese-gpt2-medium --prompt "電子計算機とは、"
```

絶対パスで指定するケース（他ディレクトリから実行する場合）:

```powershell
# ファインチューン後
C:\Users\yoshi\work\llm\.venv\Scripts\python.exe C:\Users\yoshi\work\llm\aozora\main.py generate `
    --model C:\Users\yoshi\work\llm\aozora\checkpoints\meiji_gpt2_v9 `
    --prompt "電子計算機とは、"

# ベース（比較用）
C:\Users\yoshi\work\llm\.venv\Scripts\python.exe C:\Users\yoshi\work\llm\aozora\main.py generate `
    --model rinna/japanese-gpt2-medium `
    --prompt "電子計算機とは、"
```

> `--model` は `--weights` のエイリアス。「ローカルのチェックポイントパス」と
> 「HuggingFace Hub のモデル ID」の両方を受け付けます。

サンプリング制御の推奨値:

- `--temperature 0.6〜0.7`（低めほど崩壊減）
- `--top-p 0.9` で尾部カット
- `--repetition-penalty 1.2` と `--no-repeat-ngram-size 3` で反復を抑制

> **Copilot ユーザー向け**: VS Code チャットで `/setup-aozora` と入力すると、
> [../.github/prompts/setup-aozora.prompt.md](../.github/prompts/setup-aozora.prompt.md)
> が呼ばれ、エージェントが上記 1〜4 を順に実行します。

### 観察

- **取り込めたもの**: 文末辞「〜なり／〜べし」、漢文訓読調の助詞連続、福沢・鷗外の論説構文（「〜なれば〜すべし」「ゆえに〜」「右人いかに〜」）
- **崩壊している部分**: 助詞の脱落、主語の省略、長い節での係り受け乱れ（v4 より更に減少し、節単位で論証になる）
- **率直な評価**: 5M 字 × 12 epoch + stride 256 (実効データ量 2倍) + LR 3e-5 + cosine/warmup/clip で**「複文レベルで論理が繋がる明治文」**まで到達。EMA loss 8.5 → 2.2 まで線形的に低下

## 学習設定

| 項目 | 値 |
| --- | --- |
| ベースモデル | `rinna/japanese-gpt2-medium` (336M, GPT-2 medium, T5Tokenizer SentencePiece) |
| コーパス | 5,075,448 字の連結テキスト（8 作家 36 作品） |
| ウィンドウ | 512 tokens, stride 256（50% オーバーラップ）、約 13,000 windows |
| バッチ | 2 × grad_accum 4 = 実効 8 → 約 6,500 batches / epoch |
| オプティマイザ | AdamW, peak lr=3e-5, weight_decay=0.1 |
| LR スケジューラ | cosine decay, warmup 5%（初期 976 step で 0→peak、以降緑やかに 0 へ） |
| 勾配クリッピング | `max_grad_norm=1.0` |
| エポック | 12（全 約 78,000 steps / 19,524 optim steps） |
| デバイス | 単一 RTX 5070、ピーク VRAM 約 10 GB / 240 W、約 4 時間 |

損失は 12 エポックで EMA 8.5 → 2.2 まで滞なく低下。下記 3 点の組み合わせで安定化：
1. cosine + warmup (LR が人工的に 0→peak→ 0 に推移)
2. 勾配クリッピング (値を 1.0 以内に押さえてスパイク回避)
3. stride 256 (境界文脈を見せて実効データ量 2 倍)

## ファイル構成

- [download.py](download.py) — 青空文庫作家ページ（UTF-8）から作品 zip を辿って取得＆ Shift-JIS → UTF-8 変換
- [clean.py](clean.py) — 青空文庫マークアップ（ルビ・傍点・底本情報・区切り線）を正規表現で除去
- [prepare_data.py](prepare_data.py) — 作家別比率で `data/combined.txt` を生成
- [dataset.py](dataset.py) — 512 トークンのスライディングウィンドウ Dataset（T5Tokenizer, `add_special_tokens=False`）
- [finetune.py](finetune.py) — Trainer を使わない素の PyTorch 学習ループ。`model(input_ids, labels=target_ids)` で CE 損失内部計算
- [generate.py](generate.py) — `model.generate()` の薄いラッパ（temperature / top_k / top_p / repetition_penalty / no_repeat_ngram_size）
- [main.py](main.py) — サブコマンド CLI（`download` / `clean` / `prepare` / `data` / `finetune` / `generate`）

## 改善の余地

- **さらにコーパス拡張**: 島崎藤村・新島襄・大石常心、さらに『癒学鉱言』『落葉集』等の明治期百科類も効くはず
- **アーキテクチャ変更**: rinna-medium の代わりに Llama/Qwen の 1-3B クラスを LoRA で調整
- **early stopping**: hold-out を切って過学習を監視（現状は訓練 loss のみ）
- **サンプリング温度を下げる**: 保守的に書かせたい場合 `--temperature 0.5 --top-p 0.85` まで絞る

### 試して失敗した設定

本リポジトリには `--bf16` と `--label-smoothing` フラグがあるが、
**このタスクではどちらも品質を下げた**ので既定では使わない。

- `--bf16`: 学習は 1.7 倍速くなり VRAM も半減するが、生成文が**現代論説調**に偏る。
  bf16 の 7bit mantissa では明治期の低頻度語彙に対する勾配情報が埋没し、
  文体適応が進まないと推測（v7/v8 で検証、EMA loss は改善するも生成品質は v5 より劣化）。
- `--label-smoothing 0.05〜0.1`: 出力分布を均らす正則化だが、
  明治文体という「狭い分布」への適合を阻害する方向に働き、生成文がベース rinna 寄りに戻る。
  フラグ自体は残してあるが、本タスクでは fp32 + CE 素のまま（v5 設定）が最良。
